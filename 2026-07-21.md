## KMiri 简介

它是一个面向 Rust OS kernel 的、改造自 Miri 的 Undefined Behavior 检测执行环境，目标是在尽量不改变内核开发方式的前提下，让内核代码能以接近“整核运行”的方式接受 Miri 风格的解释执行和未定义行为检查。

理想状态：除了与硬件交互的部分，在 KMiri 上运行内核，就像在用户态 Miri 上运行应用程序。

最低限度：集成进星绽日常 CI，运行和检查大多数 Rust 测试。

原版 Miri 主要面向普通 Rust 应用：它假设程序运行在一个已有的宿主 OS 环境上，通过 shim 模拟文件、线程、环境变量、socket、mmap 等常见系统服务。

但 Rust OS kernel 不是普通 `std` 程序，也不只是纯 `no_std` 程序，而是“特权代码”：

- 会直接操作页表、物理内存、CPU 寄存器等底层资源；
- 会自己建立执行环境，而不是依赖 libc 或宿主 OS；
- 会包含 inline assembly 或等价的架构相关操作。

这些行为会让原版 Miri 直接终止、无法解释，或者产生误报。KMiri 的定位就是补上这部分能力，使 Miri 能覆盖更大范围的 Rust OS 代码。

核心思路：

- 在 MIR 伪 CPU 架构层面解释执行 Rust 代码
- 非 Rust 代码的部分，通过 shim 替换
- 基于伪物理内存，并按页管理

KMiri 不是重写一个解释器，而是在 Miri 基础上扩展内存和架构模型。

因此它仍然保留 Miri 原有能力，例如检测：

- use-after-free；
- buffer overflow；
- misaligned access；
- uninitialized access；
- mutation of immutable memory；
- data race；
- borrow/provenance

也就是说，对普通 Rust typed memory 的检查能力仍来自 Miri 原有机制。

KMiri 相比原版 Miri 的关键增强在于：它可以检测一些只有 OS/kernel 环境中才有意义的 UB 或安全错误。

文档中列出的扩展包括：

- **页表相关 UB**
  - typed memory page 被错误映射到 user space；
  - 一个 typed memory page 拥有多个 active kernel virtual address；
  - 页表配置错误；
  - 页表映射与内存状态不一致。

- **untyped memory 相关 UB**
  - 对 untyped page 的 use-after-free；
  - buffer overflow；
  - misaligned access；
  - 非 volatile 访问；
  - 非 POD 类型访问。

- **页管理相关 UB**
  - 访问 reserved physical page；
  - 访问 free physical page；
  - 重复分配页面；
  - 对未分配页面执行类型转换或释放。

这些是普通 Miri 无法理解的，因为普通 Miri 不知道物理页、页表、内核虚拟地址这些概念。

Page state tracking 是 KMiri 的关键机制。

KMiri 的一个独特能力是 **page state tracking**。

它要求 OS 在访问某个物理页之前，显式告诉 KMiri 这个页当前被转换成了什么状态，例如：

- 空闲页；
- 已分配页；
- typed memory；
- untyped memory；
- page table page；
- kernel stack；
- reserved memory 等。

之后，所有通过 MIR 执行的 load/store 都会根据页面状态进行验证。

这让 KMiri 不只是检查“这个 Rust 指针是否合法”，还可以检查：

- 这个地址对应的物理页是不是允许被访问；
- 当前访问方式是否符合该页的状态；
- typed/untyped/page table 的语义有没有混用；
- 页表映射是否破坏了内核内存安全约束。

这是 KMiri 区别于普通 Miri 的核心价值之一。

原版 Miri 有自己的线程模拟系统，但它主要通过 Unix/Windows 等宿主 OS shim 来理解线程创建和调度。
KMiri 面向任意 Rust OS，不可能为每个 OS 写专门 shim，因此在适配内核线程管理方面，提供了一组通用线程管理 API，让内核主动告知 KMiri：

- 创建了哪个线程；
- 线程入口函数是什么；
- 线程栈在哪里；
- 当前切换到了哪个线程；
- 哪个 CPU 初始化了哪个线程。

KMiri 还会为每个线程维护独立的 stack space，使 stack allocation 能落在对应线程的内核栈区域，而不是普通 Miri 默认的线性 allocation 空间。

KMiri 支持 SMP 模拟，原因有两个：

- OS 中经常有多核相关逻辑；
- OS 会使用 CPU-local variables；
- 多核随机切换可以弥补原版 Miri 随机线程调度机制被禁用后的 race 检测能力。

由于 Miri 解释发生在链接前，无法直接知道 `.cpu_local` section 的真实地址，所以 KMiri 在伪物理内存的 kernel code area 中预留 CPU-local 区域。当识别到变量应链接到 `.cpu_local` section 时，就把它放到预留区域中。

在执行模型上：

- 每个 CPU core 在任意时刻绑定一个线程；
- 同一 CPU 上的线程切换只在执行调度相关代码时发生；
- KMiri 会概率性地在不同 CPU core 之间做随机上下文切换；
- 这样仍然可以触发潜在 data race。

提供面向 OS 的 shim API：KMiri 通过 shim API 让 OS 与解释器沟通。这些 API 
的作用不是给内核提供真实硬件能力，而是让 KMiri 能理解内核正在做什么，从而对后续内存访问进行正确解释和检查。

对星绽的价值：

- 总共超过 340 个 `unsafe` blocks；
- 约 220 个，也就是 65%，位于架构无关部分；
- `mm/` 是架构无关代码中 `unsafe` 最集中的模块；
- `mm/page_table/` 又是二级模块中 `unsafe` 最集中的部分。

这些模块通常已经有单元测试，并且会在真实硬件上跑。加入 Mirch 支持后，这些测试可以复用，并在 KMiri 下执行，从而获得额外的 UB 检测层。

换句话说，KMiri 的实用价值在于：它不是要求开发者重写一套 formal verification 证明，而是尽量复用已有 Rust 测试，让已有测试在更强的 UB 检测环境中运行。

不是 QEMU，也不是完整硬件模拟器。KMiri 的限制也很明确：

- 不能检查真实 CPU 架构特有行为；
- 不能解释或验证 assembly 代码中的 bug；
- 不是 QEMU，不模拟外设；
- 需要设备交互的 kernel-mode tests 无法直接运行；
- 只能检测被具体执行路径触发的 UB；
- 如果测试没有覆盖到某个 bug，KMiri 也不会发现它。

因此 KMiri 更准确的定位是：**针对 Rust OS 内核的 UB 检测增强版 Miri**，而不是完整的内核模拟器、形式化验证器或硬件 emulator。

**15. 总体定位**

一句话总结：

**KMiri 是一个将 Miri 扩展到 Rust OS kernel 场景的动态 UB 检测工具。它通过 Mirch 伪架构、伪物理内存、页表模拟、页面状态跟踪、线程/SMP/CPU-local 支持，让内核代码能在解释执行中暴露 typed memory、untyped memory、页表和页管理层面的未定义行为。**

更精炼地说，它的特点是：

- **基于 Miri**：继承 Rust MIR 解释执行和 UB 检测能力；
- **面向内核**：支持 privileged Rust OS code；
- **引入 Mirch**：用极简伪 CPU 架构替代真实硬件架构复杂性；
- **支持伪物理内存**：让 OS 可以按页管理内存；
- **支持页表**：能根据 active page table 做虚拟地址到物理地址转换；
- **支持 page state tracking**：能检查页状态与访问行为是否一致；
- **扩展 UB 覆盖面**：能发现页表、untyped memory、页管理相关 UB；
- **支持线程和 SMP**：能配合内核调度与多核逻辑检测 race；
- **复用现有测试**：适合接入已有 unit tests 和 CI；
- **工程上务实**：比形式化验证更容易落地，但不提供绝对正确性保证。
