# 星绽 KMiri 进展 - 通明湖中心	

时间：2026 年 5 月 6 日 - 7 月 21 日

## KMiri 目标

KMiri = Kernel on Miri (mid-level intermediate representation interpreter)

它是一个面向 Rust OS kernel 的、改造自 Miri 的 Undefined Behavior 检测执行环境，目标是在尽量不改变内核开发方式的前提下，让内核代码能以接近“整核运行”的方式接受 Miri 风格的解释执行和未定义行为检查。

理想状态：除了与硬件交互的部分，在 KMiri 上运行内核，就像在用户态 Miri 上运行应用程序。

最低限度：集成进星绽日常 CI，运行和检查大多数 Rust 测试（第一阶段支持 OSTD 的测试）。

## Why: 开发 KMiri 的动机

1. 星绽内核的不安全代码 （ostd 及其依赖项）必须保证不存在未定义行为
  * Miri 是 Rust 官方提供的检测语言级别未定义行为的唯一工具，但只限于用户态程序，因此需要根据内核拓展 Miri
2. Miri 具有局限性：
  * 不假设任何内存地址布局，只通过单一虚拟内存进行线性分配来保证指针不重叠；但内核依赖页表，需要考虑物理内存布局，并处理多虚拟地址空间
  * 不支持运行汇编代码
  * 只支持线程并发检测，不支持多核、多进程并发检测
3. 形式化验证代码无法跟上系统代码的开发速度，而 KMiri 在工程上务实，直接复用星绽的测试代码，并可作为 Qemu 测试的补充

对星绽的价值：

- 总共超过 340 个 `unsafe` blocks；
- 约 220 个，也就是 65%，位于架构无关部分；
- `mm/` 是架构无关代码中 `unsafe` 最集中的模块；
- `mm/page_table/` 又是二级模块中 `unsafe` 最集中的部分。

这些模块通常已经有单元测试，并且会在真实硬件上跑。加入 Mirch 支持后，这些测试可以复用，并在 KMiri 下执行，从而获得额外的 UB 检测层。

## How: KMiri 如何拓展到内核检查

KMiri 不是重写一个解释器，而是在 Miri 基础上扩展内存和架构模型，因此它仍然保留 Miri 原有能力。核心思路：

- 无法直接解释的代码，或者和解释器交互，需要通过 shims 接口
- 划分伪物理内存，并按页管理，跟踪页表的状态
- 预留 `.cpu_local` 内存区域，并把线程调度拓展到多核调度

KMiri 相比原版 Miri 的关键增强在于：它可以检测一些只有 OS/kernel 环境中才有意义的 UB 或安全错误。

### shims 是 OS 和解释器的交互接口

KMiri 提供面向 OS 的 shim API：KMiri 通过 shim API 让 OS 与解释器沟通。这些 API 
的作用不是给内核提供真实硬件能力，而是让 KMiri 能理解内核正在做什么，从而对后续内存访问进行正确解释和检查。

### 伪物理内存地址布局

![](https://hackmd.io/_uploads/HJowxhRoyg.png)

新的内存系统在 Miri 中预留一块内存区域作为伪物理内存 ，并以页级方式对其进行管理。基于该伪物理内存，KernMiri 执行更严格的地址分配。

```rust
/// Creates a default physical memory configuration with typical values:
///
/// |<-------------------------------Kernel Code Section-------------------------------->|
/// |<-Boot PT->|<-Kernel Static Section->|<-CPU-local Section->|<-Kernel Stack Section->|
/// |-----------|-------------------------|---------------------|------------------------|
/// 0x0      0x1_0000                 0x40_0000             0x41_0000                0x100_0000
///
/// |<-Kernel Code Section->|<-----Free Pages------>|
/// |-----------------------|-----------------------|
/// 0x0                 0x100_0000             0x800_0000
pub const fn default() -> Self {
    Self {
        total_mem_size: 0x800_0000,
        kernel_code_size: 0x100_0000,
        kernel_static_size: 0x3f_0000,
        cpu_local_size: 0x1_0000,

        // Size and base should align to 0x4000_0000
        kernel_code_base_vaddr: 0xffff_ffff_8000_0000,
        boot_pt_linear_mapping_base_vaddr: 0xffff_8000_0000_0000,
    }
}
```

### 分页管理 shims

通过提供的 shims API，可以将指定的物理页转换为类型化内存，从而允许在伪物理内存中为类型化对象创建 Allocation。

```rust
/// Informs KernMiri to alloc `page_count` pages at `paddr`.
///
/// Kernel should not allocate a page at the same address twice.
/// If an address that has already been allocated is allocated again
/// before being deallocated, KernMiri will treat it as UB.
pub fn kern_miri_alloc_pages(paddr: usize, page_count: usize);

/// Informs KernMiri to dealloc `page_count` pages at `paddr`.
///
/// The kernel should only deallocate an address that
/// has already been allocated by `kern_miri_alloc_pages`;
/// otherwise, such behavior will be considered UB.
pub fn kern_miri_dealloc_pages(paddr: usize, page_count: usize);

/// Informs KernMiri to retype `page_count` pages at `paddr` to `page_type`.
/// Each slot in the page has both size and alignment being `slot_size`.
///
/// After retyping, the page will become a typed page,
/// and the memory on the page will be converted into
/// a contiguous series of slots of the same size.
/// The kernel can only retype a page that has been allocated,
/// and the same page cannot be retyped twice, otherwise,
/// it will be considered UB.
pub fn kern_miri_retype_pages( paddr: usize, page_count: usize, page_type: PageType, slot_size: usize);

/// Zero `page_count` of pages out. The first page starts at `paddr`.
pub fn kern_miri_zero(paddr: usize, page_count: usize);

/// Activates the page table rooted at physical address `root_paddr` as the current page table.
pub fn kern_miri_set_root_page_table(root_paddr: usize);

/// Retrieves the root physical address of the currently active page table.
pub fn kern_miri_get_root_page_table() -> usize;
```

### 页面状态跟踪

在出于特定目的访问物理页面之前，将页面标记为特定的状态，从而检测页面数据的访问是否合法。

页面状态跟踪以及页面方案使得 KernMiri 能够检测到比 Miri 更多的 UB 类型，包括环境级 UB，例如存在缺陷或配置错误的页表。

| UB                                                                                          | Miri | KMiri |
|---------------------------------------------------------------------------------------------|:----:|:-----:|
| typed memory 释放后使用、缓冲区溢出、未对齐访问、未初始化访问、不可变对象被修改以及数据竞争 |  ✅  |   ✅  |
| 页表：访问用途与页表状态不相符、错误映射的用户空间、内核栈越界                              |  ❌  |   ✅  |
| 外设：MMIO / PIO 非法访问、非易失性访问                                                     |  ❌  |   ✅  |

### SMP 模拟

KMiri 支持 SMP 模拟，原因有两个：

- OS 中经常有多核相关逻辑；
- OS 会使用 CPU-local variables；
- 多核随机切换可以弥补原版 Miri 随机线程调度机制被禁用后的 race 检测能力。

由于 Miri 解释发生在链接前，无法直接知道 `.cpu_local` section 的真实地址，所以 KMiri 在伪物理内存的 kernel code area 中预留 CPU-local 区域。当识别到变量应链接到 `.cpu_local` section 时，就把它放到预留区域中。

在执行模型上：

- 每个 CPU core 在任意时刻绑定一个线程；
- 同一 CPU 上的线程切换只在执行调度相关代码时发生；
- KMiri 会概率性地在不同 CPU core 之间做随机上下文切换；
- 这样仍然可以触发潜在 data race。

## KMiri 的限制

KMiri 不是 QEMU，也不是完整硬件模拟器：

- 硬件行为：不能检查真实 CPU 架构特有行为、不模拟外设：
  * 不能解释或验证 assembly 代码中的 bug；
  * 需要设备交互的 kernel-mode tests 无法直接运行；
- 动态执行：
  * 只能检测被具体执行路径触发的 UB
  * 如果测试没有覆盖到某个 bug，KMiri 也不会发现它（Fuzzing 会有帮助）。
  * 并发问题依赖于随机顺序触发

通过嵌入 CPU/设备模拟器或许可以满足某些平台相关的测例。

通过各种测试技术 (Fuzzing/Mutation/Property testing) 可以覆盖更多路径和情况，从而发现更隐秘的 bug（尤其是别名问题）。

适配 GenMc (Generic Model Checker for Concurrent Programs) 并发检测器之后，可以彻底地 (exhausively) 查找并发问题。

## Miri 介绍

### Miri 在 Rust 生态中的地位

开发时间： 2015 年 11 月至今（超过 10 年）

Miri 是 Rust 官方维护的一个 MIR（中级中间表示）解释器，主要用于检测 Rust 代码中的语言级别的未定义行为 (Undefined Behavior)。

目前验证 unsafe 代码是否符合 Rust 内存模型要求的最权威工具，推动 Rust 语言内存模型走向成熟和标准化的核心力量。

* Miri 实现 Rust 内存模型（尤其 aliasing model）的可执行规范，并推动相关理论的发展
  * 为了精确定义指针和借用的生命周期，学术界和 Rust 官方团队在 Miri 中实验并实现了 Stacked Borrows ([POPL'19]) 以及更新的 Tree Borrows ([PLDI'25]) 等内存模型
  * 这些模型直接塑造了未来 Rust 官方规范对内存安全和 Aliasing rules 的定义
* 深度融入 Rust 生态的生产实践
  * [miri-test-libstd] Rust 标准库的 CI 流程中会定期使用 Miri 运行测试，以确保标准库的核心数据结构（如 Vec、HashMap、Rc 等）没有隐蔽的 UB
  * 第三方库的标配：生态中最顶尖、最基础的开源库，都会在 CI 流程中加入 `cargo miri test`，这是保证高并发、高性能底层库质量的重要手段
* Rust 语言项目官方认可
  * Miri 随 Rust 编译器工具链分发，成为 Rust 项目单独的团队
  * 其作者 [Ralf Jung][ralf] (ETH Zurich, 苏黎世联邦理工学院) Assistant Professor 是 Rust 语言项目的核心开发人员
    * miri team lead, opsem team lead, wg-const-eval lead, compiler team member

如果说 Rust 编译器是日常开发中保障安全的第一道防线，那么 Miri 就是保障底层库和 unsafe 代码安全的最后一道闸门。

### Miri 的检测能力

POPL'26 《[Miri: Practical Undefined Behavior Detection for Rust][miri-paper]》

[miri-paper]: https://dl.acm.org/doi/10.1145/3776690

* Uninitialized memory
* Temporal and spatial memory safety violations
  * out-of-bounds
  * use-after-free accesses
* Memory leaks
* Data races
  * uses a pseudo-random number generator with a fixed seed to resolve
  non-deterministic choices, and it supports a mode where the program is
  executed with many different seeds concurrently. In this mode, Miri finds
  this data race for around 90% of the executions.
* Weak memory behaviors
  * limitation: relaxed accesses do not have to be observed in a consistent
  order by all threads, so non-deterministic bugs may need many different seeds
  to be found.
* Rust type invariants
  * like invalid bit pattern in bool
* Pointer provenance
  * provenance: every pointer has exactly one allocation that it may access,
  and using that pointer to do a read or write outside the bounds of that
  allocation is UB, even if that access happens to be inbounds of another
  allocation. This extra information attached to a pointer is called provenance
* Aliasing violations
  * violating the aliasing requirements of Rust’s reference types

* 前沿并发问题检测：
  * 实验性地集成被形式化验证的 [GenMc]（弱内存模型 model checker）;[Marrying Miri and GenMC][miri-genmc]
* 形式化方向：
  *  MiniRust = idealized Miri
    * MiniRust 被设计为 Rust 核心语义的形式化规范，精确地定义 Rust 程序在执行时可能出现的行为（包括 unsafe Rust）

      支持，用以规范并证明整个 Rust 标准库
